
# Застосування технік рефакторингу

Описані 10 технік рефакторингу, які були застосовані до вихідного коду для покращення його якості, читабельності та підтримуваності.

---

## 1. Видобування методу (Extract Method)

**Опис:** Групування фрагмента коду в окрему функцію або метод, назва якого пояснює призначення цього коду.

**Фрагмент коду:**

*До:*
```python
# Всередині великої функції
if total > 1000:
    total = total * 0.9
elif total > 500:
    total = total * 0.95

```

*Після:*

```python
def calculate_discounted_total(items: List[OrderItem]) -> float:
    total = sum(item.price * item.qty for item in items)
    if total > DISCOUNT_TIER_2_THRESHOLD:
        return total * DISCOUNT_TIER_2_RATE
    if total > DISCOUNT_TIER_1_THRESHOLD:
        return total * DISCOUNT_TIER_1_RATE
    return total

```

**Пояснення:** Оригінальна функція `p_o` (`process_order`) була "Божественним об'єктом" (God Method), яка робила занадто багато речей одночасно. Розрахунок знижок — це окрема бізнес-логіка, яку варто ізолювати.
**Очікуваний ефект:** Покращення читабельності основної функції. Логіку розрахунку знижок тепер можна легко покрити юніт-тестами окремо від веб-фреймворку.

## 2. Заміна магічних чисел символічними константами (Replace Magic Number with Symbolic Constant)

**Опис:** Заміна числових або текстових літералів, які не мають очевидного змісту, на іменовані константи.

**Фрагмент коду:**

*До:*

```python
if total > 1000: ...
shipping = 50
if data["city"] == "Kyiv": 
    shipping = 0

```

*Після:*

```python
DISCOUNT_TIER_2_THRESHOLD = 1000
STANDARD_SHIPPING_COST = 50
FREE_SHIPPING_CITY = "Kyiv"

```

**Пояснення:** Використання чисел на кшталт `1000` або `50` безпосередньо в коді ускладнює його розуміння (що це за числа?) і зміну (доводиться шукати їх по всьому проєкту).
**Очікуваний ефект:** Код стає самодокументованим. Змінити тарифи на доставку або пороги знижок тепер можна в одному місці на початку файлу.

---

## 3. Перейменування методу та змінних (Rename Method/Variable)

**Опис:** Зміна назв методів, класів або змінних так, щоб вони чітко відображали свою суть і наміри.

**Фрагмент коду:**

*До:*

```python
def p_o(data: Dict):
    ...
    t = total

```

*Після:*

```python
def process_order(request: OrderRequest):
    ...
    items_total = calculate_discounted_total(request.items)

```

**Пояснення:** Короткі або абревіатурні назви (як `p_o` чи `t`) вимагають від розробника витрачати час на розшифровку їхнього значення в контексті.
**Очікуваний ефект:** Читання коду стає схожим на читання звичайної англійської мови, знижується когнітивне навантаження.

---

## 4. Введення об'єкта параметра (Introduce Parameter Object)

**Опис:** Заміна набору розрізнених параметрів (або загального словника) на спеціально створений об'єкт.

**Фрагмент коду:**

*До:*

```python
def process_order(data: Dict): # Словник без структури

```

*Після:*

```python
class OrderRequest(BaseModel):
    user_id: int = Field(gt=0)
    items: List[OrderItem]
    address: Address

def process_order(request: OrderRequest):

```

**Пояснення:** Використання базового типу `Dict` (одержимість примітивами) позбавляє нас автодоповнення в IDE та вимагає ручної перевірки ключів (наприклад, `if "user_id" in data`).
**Очікуваний ефект:** Строга типізація, автоматична валідація даних завдяки Pydantic та надійніша архітектура бекенду.

---

## 5. Видобування класу (Extract Class)

**Опис:** Створення нового класу для групи полів і методів, які логічно пов'язані між собою і часто використовуються разом (боротьба зі "згустками даних" - Data Clumps).

**Фрагмент коду:**

*До:*

```python
if "street" in data and "city" in data and "zip" in data:
    ...

```

*Після:*

```python
class Address(BaseModel):
    street: str
    city: str
    zip_code: str = Field(alias="zip")

```

**Пояснення:** Властивості `street`, `city` та `zip` завжди йдуть разом, оскільки формують поняття "Адреса". Тримати їх окремо нераціонально.
**Очікуваний ефект:** Підвищення зв'язності (cohesion) коду. Об'єкт `Address` тепер можна легко перевикористовувати в інших частинах системи.

---

## 6. Заміна вкладених умов на Guard Clauses (Replace Nested Conditional with Guard Clauses)

**Опис:** Використання "раннього повернення" (early return) для перевірки крайніх випадків замість використання глибоко вкладених конструкцій `if-else`.

**Фрагмент коду:**

*До:*

```python
if "user_id" in data:
    if data["user_id"] > 0:
        if "items" in data and len(data["items"]) > 0:
            # Корисна логіка

```

*Після:*

```python
if not request.items:
    raise HTTPException(status_code=400, detail="No items provided")
# Корисна логіка на першому рівні вкладеності

```

**Пояснення:** Антипатерн "Arrow" (глибока вкладеність) ховає основний потік виконання програми. Guard clauses дозволяють відсіяти помилки на самому початку функції.
**Очікуваний ефект:** "Плоский" код, значне покращення читабельності та спрощення розуміння основного (щасливого) сценарію виконання.

---

## 7. Видалення мертвого коду (Remove Dead Code)

**Опис:** Вилучення коду, який більше не використовується, закоментованих блоків або непотрібних змінних.

**Фрагмент коду:**

*До:*

```python
x = 10
debug_mode = True
# print("Processing started")

```

*Після:*
*(Блок повністю видалено)*

**Пояснення:** Залишки старого коду для зневадження або невикористані змінні створюють "інформаційний шум" та можуть заплутати інших розробників.
**Очікуваний ефект:** Чистіший файл, менша кількість рядків коду, відсутність зайвих питань у команди під час рев'ю.

---

## 8. Розщеплення тимчасової змінної (Split Temporary Variable)

**Опис:** Створення окремих змінних для кожного проміжного результату, замість того, щоб перевикористовувати одну й ту саму змінну для різних цілей.

**Фрагмент коду:**

*До:*

```python
total = sum(...)
if total > 1000: total = total * 0.9 # Мутація змінної
final_total = total + shipping # Знову використання

```

*Після:*

```python
items_total = calculate_discounted_total(request.items)
shipping_cost = calculate_shipping(request.address.city)
final_total = items_total + shipping_cost

```

**Пояснення:** Змінна `total` спочатку означала суму товарів, потім суму зі знижкою. Така мутація робить код нестабільним і важким для дебагінгу.
**Очікуваний ефект:** Значення кожної змінної залишається незмінним після ініціалізації. Чітко видно, з яких компонентів складається підсумкова сума.

---

## 9. Переміщення методу (Move Method)

**Опис:** Перенесення методу або логіки туди, де вона логічно належить (наприклад, ближче до даних, з якими вона працює).

**Фрагмент коду:**

*До:*

```python
# Логіка визначення доставки була прямо всередині process_order
shipping = 50
if data["city"] == "Kyiv":
    shipping = 0

```

*Після:*

```python
def calculate_shipping(city: str) -> float:
    if city == FREE_SHIPPING_CITY:
        return 0.0
    return float(STANDARD_SHIPPING_COST)

```

**Пояснення:** Функція створення замовлення не повинна знати деталі тарифів транспортної компанії. Це порушення принципу єдиної відповідальності (SRP).
**Очікуваний ефект:** Розділення відповідальності. Функція доставки може бути розширена незалежно від логіки обробки замовлень.

## 10. Інкапсуляція колекції (Encapsulate Collection / Repository Pattern)

**Опис:** Приховування прямого доступу до структури даних за допомогою методів, які контролюють додавання або отримання елементів.

**Фрагмент коду:**

*До:*

```python
db = [] # Глобальна змінна
# ...
db.append(order) # Прямий доступ у функції

```

*Після:*

```python
class OrderRepository:
    def __init__(self):
        self._db: List[OrderResponse] = []

    def save(self, order: OrderResponse) -> None:
        self._db.append(order)

    def get_by_id(self, order_id: int) -> Optional[OrderResponse]: ...

order_repo = OrderRepository()
# ...
order_repo.save(order)

```

**Пояснення:** Використання глобального списку для зберігання стану є небезпечним. Будь-яка частина програми могла змінити `db`. Крім того, це перешкоджає майбутньому переходу на реальну базу даних.
**Очікуваний ефект:** Дані надійно інкапсульовані. Коли прийде час підключати реальну БД (наприклад, PostgreSQL через SQLAlchemy), достатньо буде змінити лише методи класу `OrderRepository`, не торкаючись бізнес-логіки ендпоінтів.
